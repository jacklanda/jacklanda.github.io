区别构造和赋值
一样的结果&不一样的行为

为了便于理解后面的内容，我们定义了一个 Widget 类。

class Widget {
public:
    Widget() = default;                       // 默认构造函数
    Widget(bool sign) : isAvailable(sign){};  // 普通构造函数
    Widget(const Widget& w) : isAvailable(w.isAvailable){};  // 拷贝构造函数
    Widget(Widget&& w) : isAvailable(std::exchange(w.isAvailable)){};  // 移动构造函数
    Widget& operator=(const Widget& w) {  // 拷贝赋值函数(操作符)
        isAvailable = w.isAvailable;
        return *this;
    }
    Widget& operator=(Widget&& w){  // 移动赋值函数(操作符)
        isAvailable = w.isAvailable;
        return *this;
    }
    ~Widget(){};  // 析构函数
private:
    bool isAvailable = false;
    char* s;
};

接着，咱们来创建几个 Widget 类对象

int main() {
    Widget w1;        // 1
    Widget w2(true);  // 2
    Widget w3(w1);    // 3
    Widget w4 = w3;   // 4
    w4 = w2;          // 5
}

那么问题来了，上面这四种对象的创建方式，他们有什么异同呢？

首先，你已经观察到我们在 Widget 类中声明并实现了7个特殊成员函数，从上到下依次是：默认构造函数，普通构造函数，拷贝构造函数，拷贝赋值函数，析构函数。

  • 第1种方式创建了新对象 w1，且在创建时没有任何的入参，调用的是默认构造函数；
  • 第2种方式根据传入的一个布尔字面量 true 创建了新对象 w2，调用的是普通构造函数；
  • 第3种方式传入了我们先前创建的 w1 对象，根据现有对象构造新对象 w3，调用了拷贝构造函数；
  • 第4种方式根据等号右边的现有对象 w3 创建了新对象 w4，同样调用了拷贝构造函数；
  • 第5种方式看似与第四种方式相同，但不难发现它其实是根据已有对象对同样是已有对象 w4 的重新赋值，因此调用的是拷贝赋值函数，而非拷贝构造函数。

怎么 C++ 创建一个对象有这么多花里胡哨的方式？ 的确，最初看到这几个语句的时候我也很疑惑，但随着学习的深入，越来越认同“C++ 给了开发者更多的自由，也给了它们更多的心智负担。”但讲到的这部分内容仅仅是给本文开了个头，暂且先带着疑惑，继续读下去吧~

这里引出了第一个知识点：构造行为和赋值行为。第3和第4条语句的行为其实是完全等效的，实际调用的都是 Widget 的拷贝构造函数 Widget(const Widget& w) 。而第5条语句则是先前已经建立了 w2 与 w4 对象，现在是将 w2 对象赋给 w4 对象，属于赋值行为。

(n.b. 至此，如果你已经十分清楚上面这些语句为什么行为不同，就不用继续跟进下面的内容了。)

Rule of three/five/zero

我们知道 C++11 是 C++ 发展历程中的一个重要版本，引入了许多现代编程语言特性。在 C++11 以前，只有 “Rule of three” 这种说法，C++11 以后，随着移动语义的引入，便扩展为 “Rule of five” 。

three 指的类内的三个特殊成员函数：拷贝构造、拷贝赋值和析构。我们上边讲的就是 Rule of three 的内容。

five 指的是在 three 的基础上引入移动语义(下文会讲)，增加了移动构造函数与移动赋值函数(操作符)，加上原来的3个一共就是5个特殊成员函数。程序员需要手动地显式声明与实现这5个特殊成员函数，因而得名 The rule of five。

zero 则站在 five 的对立面，也就是说：要么将5个特殊成员函数显式地定义清楚(显式地自定义实现 / 显式地设为默认 default / 显式地通过 delete 禁用)，实现 Rule of five；要么就 Rule of zero，干脆啥也不去定义、实现。

语义分类
Value categories

Each C++ expression (an operator with its operands, a literal, a variable name, etc.) is characterized by two independent properties: a type and a value categories. Each expression has some non-reference type, and each expression belongs to exactly one of the three primary value categories: prvalue, xvalue, and lvalue.
       —— —— cppreference/value categories

上面这段话的意思是：每个 C++ 表达式都具有两个独立的属性，一个是类型 (type)，另一个是值类别 (value categories)，如：char, int, double 这些都是常见的 type；而 value categories 则包含了3种类别：左值、纯右值与将亡值。下面将简单介绍一下这几个类别的含义。

拥有身份(glvalue) 不拥有身份
可移动 (rvalue) xvalue prvalue
不可移动 lvalue 不存在

左值(lvalue)

顾名思义左值是位于赋值符号=左边的值，准确地说，左值是在内存中有具体内存位置的值，是在表达式(e.g. 赋值表达式=)运算后依然存在的持久对象。

"hello world";  // lvalue
int num = 1055;  // lvalue
++num;  // lvalue
int* p_num = &num;  // lvalue

int& access(){ return num; }
access();  // lvalue

int arr[10]{};  // lvalue
arr[0];  // lvalue

int foo();
int&& rr_foo { foo() };  // lvalue

struct A{ int m_a{1}; };
A obj;  // lvalue
obj.m_a;  // lvalue
A* p_obj = &obj;  // lvalue
A& r_obj = obj;  // lvalue

在上例中，不论是值类型变量 num，指针类型变量 p_num，类类型值对象 obj，类类型指针对象 p_obj，左值引用对象 r_obj，右值引用对象 rr_foo，它们都属于左值。

lvalue 具有以下特征： - 所有 glvalue 具有的特征 - 可以通过取址运算符获取其地址 - 可修改的左值可用作内建赋值和内建符合赋值运算符的左操作数 - 可以用来初始化左值引用（下文会讲）

纯右值(prvalue)

在 C++11 前，value categories 只划分为了左值和右值两种类型，C++11 以后，由于引入移动语义(又是它)的需要，又将右值细分为了纯右值和将亡值。

1055;  // prvalue
true;  // prvalue

1.0 + 2.0;  // prvalue

int a{}, b{};  // both lvalues
a + b;  // prvalue
&a;  // prvalue

a++;  // prvalue
b--;  // prvalue

a && b;  // prvalue
a < b;  // prvalue

double {};  // prvalue

std::vector<_Tp> {};  // prvalue

/* 5. Lambda Expression */
[](int a, int b){return a + b;}

顾名思义，纯右值就是纯粹的右值，要么是纯粹的字面量，要么是求值结果相当于字面量或匿名临时对象。如上所示，例1和例2是原始字面量；例3属于运算表达式产生的临时变量；例4属于非引用返回的临时变量，在调用 func 函数后将返回一个值对象 obj ，这个值对象会通过拷贝构造函数在 main 函数内产生一个临时对象，这个临时对象属于纯右值；例5则是一个 Lambda 表达式，以上这些表达式在 C++ 里都称为纯右值。

将亡值(xvalue)

将亡值是即将被销毁（将亡值对象处于生命期的末尾），但却能够被移动的值。常见的将亡值有：由纯右值实体化创建的临时变量，函数内部将要被 return 的某个局部变量（在它生命的最后时刻）。

bool b {true};  // lvalue
std::move(b);  // xvalue
static_cast<bool&&>(b);  // xvalue

int&& foo();
foo();  // xvalue

struct foo { int m_a; };
std::move(foo()).m_a;  // xvalue
foo{}.m_a;  // xvalue

int arr[10]{};
std::move(arr);  // xvalue
std::move(arr)[0];  // xvalue

int[10]{}[0];  // xvalue

/* nums 属于 xvalue */
int* create(){
    int nums[] = {1, 2, 3};
    return nums; // Warning!
}
int main(){
    int* arr = create();
}

如上所示的栈数组 nums 是在 create() 内创建的局部变量，如果不采取额外的措施(e.g. 使用 static 将数组分配在静态存储区，or 使用移动语义延长对象的生命期)，编译器将抛出 Warning，因为其指向的对象将在 return 的时候被销毁(由 leave 汇编语句释放当前栈帧使用的资源)。所以在初学 C 语言时，老师反复强调的“不能返回指向当前栈帧内存的局部变量”就是这么来的。

xvalue 具有所有 glvalue 和 rvalue 拥有的特征。

左/右值引用&引用坍缩

好了，现在我们知道什么是左值、纯右值与将亡值了。接着讲一下基于左值的左值引用以及基于右值的右值引用。

左值引用

C++11 前，我们所说的引用没有歧义，指的就是对左值进行引用。

#include <iostream>
#include <type_traits>

class A {};

int main() {
    std::cout << std::boolalpha;
    std::cout << std::is_lvalue_reference<int>::value << '\n';  // 输出: false
    std::cout << std::is_lvalue_reference<int&>::value << '\n';  // 输出: true
    std::cout << std::is_lvalue_reference<A>::value << '\n';   // 输出: false
    std::cout << std::is_lvalue_reference<A&>::value << '\n';  // 输出: true
}

btw, C++ 里的左值引用其实暗含了一次对指向对象的解引用操作，原因在于 int& ↔︎ int* const，A& ↔︎ A* const 。

右值引用

传统的 C++ 没有区分『移动』和 『拷贝』的概念，造成了大量的数据拷贝，浪费时间和空间。右值引用的出现恰好就解决了这两个概念的混淆问题。  —— —— 《现代C++教程》

C++11 提出了右值引用这一概念，合理化了 int&& num = 3 这种对右值3取引用的写法。此外，C++11 提供了 std::move 方法让我们可以把左值无条件地转换为右值(viz. xvalue)。

#include <iostream>
#include <type_traits>
 
class A {};
 
int main() 
{
    std::cout << std::boolalpha;
    std::cout << std::is_rvalue_reference<A>::value << '\n';  // 输出: false
    std::cout << std::is_rvalue_reference<A&>::value << '\n';  // 输出: false
    std::cout << std::is_rvalue_reference<A&&>::value << '\n';  // 输出: true
    std::cout << std::is_rvalue_reference<int>::value << '\n';  // 输出: false
    std::cout << std::is_rvalue_reference<int&>::value << '\n';  // 输出: false
    std::cout << std::is_rvalue_reference<int&&>::value << '\n';  // 输出: true
}

引用坍缩

传统C++里规定不能对一个引用类型继续进行引用，但在右值引用出现后就允许我们对引用进行新的左/右引用，即产生了所谓的引用坍缩规则:

函数形参类型 实参参数类型 推导后函数形参类型
T& 左引用 T&
T& 右引用 T&
T&& 左引用 T&
T&& 右引用 T&&

因为引用坍缩的存在，如果我们希望入参在调用函数内部保持传入前的参数类型，则需通过 std::forward 进行完美转发，详细用法网上有很多，这里就不展开了。

++a 和 a++ 的区别？

C 语言课上，老师从使用效果上讲过 ++a 和 a++ 的区别：++a 表示变量值 a 先自增加1后返回这个值，a++ 叫后自增，表示自增 + 1 了，但返回的是自增前的变量值。但这一过程是如何是实现的呢？

现在，我们知道了左右值以及它们引用的概念，可以从代码上来一窥究竟了：

/* 前缀自增 */
int& int::operator++(){
    *this += 1;
    return *this;
}

/* 后缀自增 */
const int int::operator++(int){
    int old_val = *this;
    ++(*this);
    return old_val;
}

也许，看了上面的代码，你会对左右值以及自增运算符有了更清晰的认识。

所有权

在讲拷贝/移动语义前，笔者打算简单说一下自己对于 “所有权(Ownership)” 概念的理解。笔者最初在学 Rust 的时候接触的“所有权”这一概念，不同于 C++ 语句默认采用拷贝语义，Rust 语句默认使用移动语义，也正是由于所有权的规约，无 GC 的 Rust 一直以高效、内存安全著称。

首先明确一点，所有权针对的是堆区内存，因为栈区内存是编译期确定且随着函数栈帧的创建/释放自动管理的，而堆区内存的分配和使用只有运行时才得以确定，因此 Rust 所有权的职责说白了就是追踪堆内存数据的动向。

拷贝语义

如本文开头所举的那个例子，C++ 中的拷贝语义由拷贝构造函数和拷贝赋值操作符承担。 - 拷贝构造函数 (Copy Construction)：负责让对象被创建作为一个对现有对象的副本 的类内特殊成员函数。 - 拷贝赋值操作符 (Copy Assignment)：负责让现有对象的内容被赋值替换为另一现有对象的内容 的类内特殊成员函数。

自定义拷贝构造函数

在使用类库的时候，我们可以使用 std::vector<int> new_vec = old_vec; 这样的写法，是因为类库中的模板类 vector 已经给我们实现好了拷贝构造函数。对于我们自己的类，设计时如果用到了拷贝语义，就需要手动实现一份拷贝构造函数。

如下，假设我们需要设计一个自己的 vector 模板类。

template <typename T>
class vector{
public:
    /* copy construction */
    vector(const vector<T>& other){ 
        m_size = other.m_size;
        m_capacity = other.m_capacity;
        std::copy(other.m_elems, other.m_elems + other.m_size, m_elems);
    }
private:
    size_t m_size;
    size_t m_capacity;
    int* m_elems;
};

同理，我们也需要自己实现赋值操作符的重载。

移动语义

终于讲到移动啦！与 Rust 对左值的“直接移动”不同，C++ 的移动语义需要先通过 std::move() 将左值转换为将亡值(n.b. C++ 中只有右值可以被 move)，然后再将此将亡值赋给新的变量（借助移动构造/赋值函数），这才走完了所有权移动的过程。

上面提过的 xvalue, prvalue 以及 lvalue ，这三者遵循着一套移动规则以及对应着不同的构造方法： |Value categories|移动规则 |调用构造方法| |:—————|:——————|:—————–| |xvalue |可以被直接移动 |调用移动构造函数 | |prvale |被移动的时候，会生成一个临时变量 xvalue，这个 xvalue 被移动了 |prvalue 初始化 lvalue，直接构建，不调用拷贝/移动构造函数 | |lvalue |移动时需要使用 std::move 将 lvalue 变成 xvalue，然后再移动这个 xvalue |调用拷贝构造函数 |

发生移动后，原来的被移动的变量指向的值的所有权转移到了新的变量中，原变量变成了“空值”状态。在下面的这个例子中，我们通过 std::move 将变量 old_str 变成了一个 xvalue，并将它赋给变量 new_str，这时候将程序调用 std::string 类型的移动函数来操作与管理 old_str 指向的资源 “hello world”

std::string old_str = "hello world";
std::string new_str = std::move(old_str);  // after moved, old_str == "", new_str == "hello world";

性能优化
编译期优化

Copy elision 和 RVO/NRVO 是两套独立的优化策略。 ##### 拷贝省略 拷贝省略（Copy elision）是编译期的重要优化之一，将不需要的 copy/move 操作以及析构操作在编译期直接去除，从而减少多次操作带来的开销。我们将用一个例子说明在拷贝省略的过程中编译器究竟为我们做了哪些事情。

#include <iostream>

class A {
public:
    A() { std::cout << "Constructed\n"; }
    A(const A&) { std::cout << "Copied\n"; }
    A(A&&) { std::cout << "Moved\n"; }
};

A create() {
    A obj_create;
    return obj_create;
}

int main() {
    A obj = create();
}

不管是 C++11 还是 C++17 ，它们都把拷贝省略把移动操作(对拷贝同理，没有移动构造函数的时候省略拷贝构造函数)给省略了。那么不禁要问是在什么时候发生的以及是怎么省略的呢？

这里先给出结论：如果不发生拷贝省略，C++11 编译出的 A obj = create(); 语句将发生下面的5件事：

  1. 调用函数 create 前，使用一块内存 “obj_mem” 用来存储对象 obj；
  2. 准备进入函数 create 栈帧前，选择一块临时内存 “tmp_mem” 来接收函数的返回值；
  3. 压入 create 函数栈帧后，选择一块当前函数帧上的内存来创建局部对象 obj_create；
  4. 函数返回时，将局部对象 obj_create 移动到上个函数帧的临时内存 “tmp_mem” 上；
  5. “赋值”的时候，从 “tmp_mem” 上将 obj_create 的内容移动给“obj_mem”。

可以看到这个过程一共发生了两次移动（第四和第五步），为了进一步验证其背后的行为，我们来探索上例编译后生成的汇编代码。

生成汇编如下： create(): push rbp mov rbp, rsp sub rsp, 32 mov QWORD PTR [rbp-24], rdi lea rax, [rbp-1] mov rdi, rax call A::A() [complete object constructor] lea rdx, [rbp-1] mov rax, QWORD PTR [rbp-24] mov rsi, rdx mov rdi, rax call A::A(A&&) [complete object constructor] mov rax, QWORD PTR [rbp-24] leave ret main: push rbp mov rbp, rsp sub rsp, 16 lea rax, [rbp-1] mov rdi, rax call create() lea rdx, [rbp-1] lea rax, [rbp-2] mov rsi, rdx mov rdi, rax call A::A(A&&) [complete object constructor] mov eax, 0 leave ret

可以看到，在 C++11 无拷贝优化下，我们的代码在 create() 与 main() 里分别 call 了两次 A::A(A&&)，即两次移动构造。

使用 C++17 编译代码和运行 clang++ -std=c++17 -fno-elide-constructors main.cpp -o main && ./main ，输出：

Moved

可见，在 C++17 无拷贝优化下，我们的代码只在 create() 中调用了一次移动构造函数 A::A(A&&) 。

很神奇的发现：在对象被构造出来后，C++11 分别在 create() 和 main() 内 各自 move 了一次，C++17 只在 create() 内 move 了一次。

如果我们启用拷贝省略进行编译：g++ -std=c++11 main.cpp -o main && ./main (g++ -std=c++17 main.cpp -o main && ./main) 这时候会发生什么呢？

对应的汇编如下：

A::A() [base object constructor]:  ; 类A
        push    rbp
        mov     rbp, rsp
        sub     rsp, 16
        mov     QWORD PTR [rbp-8], rdi
        mov     esi, OFFSET FLAT:.LC0
        mov     edi, OFFSET FLAT:_ZSt4cout
        call    std::basic_ostream<char, std::char_traits<char> >& std::operator<< <std::char_traits<char> >(std::basic_ostream<char, std::char_traits<char> >&, char const*)
        nop
        leave
        ret
create():  ; create()函数入口
        push    rbp
        mov     rbp, rsp  ; 函数帧序言
        sub     rsp, 16
        mov     QWORD PTR [rbp-8], rdi  ; 将rdi内容赋给内存中的[rbp-8]处，QWORD PTR 意为解释后接内存区域[rbp-8]为4个字(8字节)的指针类型
        mov     rax, QWORD PTR [rbp-8]
        mov     rdi, rax
        call    A::A() [complete object constructor]  ; 调用类A的普通构造函数
        nop
        mov     rax, QWORD PTR [rbp-8]
        leave
        ret
main:  ; main()函数入口
        push    rbp
        mov     rbp, rsp  ; 函数帧序言
        sub     rsp, 16
        lea     rax, [rbp-1]   ; 将 [rbp-1] 内存的地址(非内存中存储的内容)赋给rax寄存器
        mov     rdi, rax
        call    create()  ; 调用create函数
        mov     eax, 0 ;  将eax寄存器内容置零
        leave
        ret

启用拷贝优化后汇编只是 call 了普通构造函数，再没有别的拷贝/移动操作了。原来，编译器默默地帮我们做了这么多事情 ٩(๑>◡<๑)۶

现在我们知道了编译器会默认帮我们开启拷贝省略，从而我们写的代码一次移动都没做。此外如果关闭了拷贝省略，就会因为 C++17 后编译器默认使用 RVO ，可选使用 NRVO ，所以只有产生了一次移动操作；而 C++11 标准下编译器默认不会帮你做这两件事情，对应产生了两次理应被优化掉的移动操作。

所以，什么是 RVO 与 NRVO 呢？

RVO 与 NRVO

RVO 是英文 return value optimization 的缩写，NRVO 是英文 named return value optimization 的缩写。

下面的这段解释来自知乎上一位网友的文章（见文末参考[8]） > RVO是返回值优化，就是将返回值的创建省略了。NRVO是函数内具有名字的局部变量充当返回值的时候，它的创建也被省略了。它们的省略结果都是在最终的变量里创建对象。

// 写法一
Verbose create() {
  return Verbose();
}
//写法二
Verbose create() {
  Verbose v;
  return v;
}
//写法三
Verbose create() {
  Verbose v;
  return std::move(v);
}

这里首推写法一，因为在使用现代编译器的前提下(C++11后)，会默认将这段代码编译成直接在主调函数帧上原地构造对象，忽略了无编译优化条件下的两次移动操作，也就是我们前文提到的 copy elision。

如果编译器支持，那么针对写法二会进行 NRVO 优化。 > 在C++没有move的时候，很多编译器实现就支持RVO(return value optimization)和NRVO(named return value optimization)。这两种优化用于在返回值的时候避免不必要的copy。在C++11以后，引入了move，move的存在与copy elision交织在一起。在这里直接给出答案，写法一是推荐的，写法二场景需要就推荐，而写法三是不推荐的。

C++17以后，RVO是保证会发生的，因为标准这么定义了。而NRVO并没有保证会发生，但是大部分编译器都会实现。所以在C++17里面，没有copy elision的时候只有一个moved，因为RVO也存在，而这个moved，是由于NRVO没有开启生成的。如果开启优化，那么一个moved也没有，因为NRVO把这个moved给优化掉了。当用prvalue创建对象的时候，直接会构造对象，而不用调用那几个构造函数。

开发层优化

入参主动转换为右值

设置函数形参为右值引用，入参强制左值转换为右值，这样可以直接降低入参拷贝产生的性能开销。

#include <iostream>
#include <vector>

void take(std::vector<int>& param_vec) {
    for (int& n : vec) std::cout << n << ','; // Output: 6,6,6
    std::cout << '\n';
}

void take(std::vector<int>&& param_vec) {
    for (int& n : vec) std::cout << n << ','; // Output: 6,6,6
    std::cout << '\n';
}

int main() {
    std::vector<int> vec(3, 6);
    take(vec);  // 入参为左值引用
    take(std::move(vec));  // 入参为右值引用
    std::cout << "----------------------------------------\n";
    for (int& n : vec) std::cout << n << ',';  // Output nothing.

对于上面的代码段，左值 vec 在 move() 作用下转变为了一个 xvalue 值，并作为函数 take() 的入参，从输出的结果上看，调用 move() 函数前左值 vec 拥有 vector对象 {6, 6, 6} 的所有权，调用 move() 后， vector 对象所有权转移给了 take() 函数的形参 param_vec ，变量 vec 也就不再拥有对原 vector 对象的所有权了。
